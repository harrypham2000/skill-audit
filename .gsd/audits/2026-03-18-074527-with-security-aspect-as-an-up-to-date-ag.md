## Strengths
- **Direct OWASP Agentic Top 10 Alignment**: Finding IDs in `src/audit.ts` map directly to ASIxx categories (e.g., ASI01 for Goal Hijacking, ASI04 for Supply Chain).
- **Advanced Dependency Scanning**: The `src/deps.ts` implementation is exceptionally robust, supporting multiple scanners (Trivy, OSV CLI) and a fallback to the OSV.dev API via `curl` for environments without pre-installed security tools.
- **Risk-Based Scoring Model**: Implements a weighted 0-10 risk scoring system in `src/scoring.ts` that considers both finding severity and category importance (e.g., Supply Chain risks weigh more heavily than Behavioral warnings).
- **Multi-Agent Awareness**: Seamlessly integrates with the Vercel `skills` CLI via `src/discover.ts`, allowing it to audit skills across multiple agents (Claude Code, Cursor, Codex) in both global and project scopes.
- **Path Traversal Protection**: Discovery logic in `src/discover.ts` includes boundary checks for symlinks to prevent malicious skills from escaping their directory via symbolic links.
- **Comprehensive Pattern Library**: Includes detection for prompt injection (role-play, instructions override), credential leaks (SSH/AWS paths), exfiltration chains (File Read + Send), and dangerous code execution (`rm -rf`, `eval`, `subprocess.shell=True`).

## Gaps
- **Static Analysis Limitations**: The auditor relies exclusively on regex-based static analysis. It lacks semantic analysis or LLM-powered inspection for complex or obfuscated malicious behaviors mentioned in the roadmap.
- **Hardcoded Rule Definitions**: Security patterns and rules are hardcoded within `src/audit.ts` rather than being externalized into the `rules/` directory, making updates and community contributions more difficult.
- **Missing CI/CD Output Formats**: While the tool supports JSON and Console output, it lacks SARIF (Static Analysis Results Interchange Format) support, which is critical for integration with GitHub Actions and GitLab CI security dashboards.
- **Reactive vs. Proactive Auditing**: The current implementation primarily audits *installed* skills. It is not yet integrated as a pre-install hook for `npx skills add`, which is a primary goal of the ecosystem's security plan.
- **Incomplete OWASP Mapping**: Coverage is strong for ASI01, ASI02, ASI04, ASI05, and ASI09, but other categories like ASI03 (Planning Strategy Manipulation) and ASI07 (Insecure Output Handling) are missing specific detection patterns.

## Next Steps
1. **Implement SARIF Reporter**: Add a new reporter mode to `src/reporter.ts` to emit SARIF JSON for seamless CI/CD integration.
2. **Externalize Security Rules**: Migrate hardcoded regex patterns from `src/audit.ts` to YAML/JSON files in the `rules/` directory to facilitate "security-as-code" updates.
3. **Expand Detection Patterns**: Research and implement specific regex patterns for the remaining OWASP Agentic Top 10 categories, particularly ASI03 and ASI07.
4. **Develop Pre-Install Hook**: Create a wrapper or integration logic to allow the auditor to run *before* `skills add` completes, providing a proactive security gate.
5. **Add Semantic Analyzer**: Investigate integrating a lightweight LLM-based scanner for high-confidence classification of complex instruction intent (e.g., exfiltration vs. legitimate backup).
